# Auth API Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use @superpowers/subagent-driven-development (recommended) or @superpowers/executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking. **Controllers** implementing with Task/subagents: for any Prisma schema/migrate work, follow `.cursor/rules/subagent-prisma-host-handoff.mdc` and `.cursor/skills/prisma-delegate-migrate/SKILL.md` (host runs `migrate dev` after `NEEDS_HOST_PRISMA`).

**Goal:** Implement first-party email/password registration (create household vs join via invitation), login, signed JWT access tokens in production, authenticated logout without household tenancy, household invitation issuance, `HouseholdMembership` roles (on **`GET /api/me`**, **`GET /api/household/members`**, and tenancy), `dev-seed`, and OpenAPI/docs alignment per [`../specs/2026-03-28-auth-api-design.md`](../specs/2026-03-28-auth-api-design.md).

**Architecture:** Extend Prisma with `User.email` / `User.passwordHash`, `HouseholdMembership.role`, and `HouseholdInvitation`. Add focused modules: **email normalization**, **password hashing** (Argon2id via a Bun-compatible library), **JWT sign/verify** (HS256 via `jose` using `JWT_SECRET` + `JWT_EXPIRES_IN`). Refactor `createFetchHandler` in `src/api/server.ts` so **`POST /api/auth/register`** and **`POST /api/auth/login`** bypass auth and tenancy; **`POST /api/auth/logout`** resolves identity only (same as protected routes) and **skips** `getHouseholdForUser`; all other `/api/*` routes keep today’s flow. Put register/login/invite transactions in **`src/api/services/auth-service.ts`** and **`src/api/services/invitations-service.ts`** (or a single `auth-invitations-service.ts` if you prefer one file—keep handlers thin). Replace decode-only JWT behavior in `src/api/auth.ts` with **verified** tokens in non-development mode. **`GET /api/household/members`** returns **`role` per row** once `HouseholdRole` exists (Task 9).

**Tech Stack:** Bun, TypeScript, Prisma, PostgreSQL, `jose` (HS256), Argon2id (`@node-rs/argon2` or `argon2`—pick one that installs cleanly on your machine; verify with a unit test), existing `src/api/errors.ts`, `ApiProblem`, `readJsonBody` patterns.

**Spec anchors:** HTTP status codes and error codes follow the auth spec; `401` uses stable codes such as `invalid_token` / `invalid_credentials` where specified.

**Files:**

**Schema sketch (adjust relation names to Prisma conventions):**

**`User.email`:** The spec requires **unique, required** email for locally registered users ([spec § Data model / User](../specs/2026-03-28-auth-api-design.md)). Use **`email String @unique`** for greenfield (all `User.create` paths, including `tests/integration/api/helpers.ts` `seedHouseholdUser`, must supply a normalized email). If a dev database already has rows without email, follow the spec’s migration/backfill note: backfill or reset before applying NOT NULL—**do not** hand-edit migration SQL; use Prisma’s workflow (e.g. optional column + data fix + follow-up migrate, or `migrate reset` in local dev only).

```prisma
enum HouseholdRole {
  OWNER
  MEMBER
}

model User {
  // existing...
  email         String @unique
  passwordHash  String?
  // relations: invitationsCreated HouseholdInvitation[] @relation("InvitationCreatedBy")
  // invitationsUsed   HouseholdInvitation[] @relation("InvitationUsedBy")
}

model HouseholdMembership {
  userId      String
  householdId String
  role        HouseholdRole @default(MEMBER)
  // @@id unchanged
}

model HouseholdInvitation {
  id              String    @id @default(uuid())
  householdId     String
  email           String
  tokenHash       String    @unique
  createdByUserId String
  expiresAt       DateTime
  usedAt          DateTime?
  usedByUserId    String?
  createdAt       DateTime  @default(now())

  household   Household @relation(fields: [householdId], references: [id], onDelete: Cascade)
  createdBy   User      @relation("InvitationCreatedBy", fields: [createdByUserId], references: [id], onDelete: Restrict)
  usedBy      User?     @relation("InvitationUsedBy", fields: [usedByUserId], references: [id], onDelete: SetNull)
}
```

### Task 6: Auth service — register (create household) and login

**Files:**

- Create: `src/api/services/auth-service.ts`
- Modify: `src/domain/dtos/user.ts` (or new `src/domain/dtos/auth.ts`) for JSON shapes

**Behavior (see spec § `POST /api/auth/register`, `POST /api/auth/login`):**

- **HTTP:** Register success → **`201 Created`**; login success → **`200 OK`**; both return the same JSON envelope (access token + user). Logout → **`204`** (Task 7).
- Optional **`householdName`** on create-household path: if omitted, default household name **`"Home"`** (document in OpenAPI).
- On create-household path, after transaction the household has **exactly one** `HouseholdMembership` with **`role = OWNER`** (spec invariant: one OWNER per household, enforced in app logic on create/invite flows).
- Password min length **10**; weak → `422` + code `validation_error` or dedicated code per OpenAPI.
- Ambiguous body (both `householdName` and `householdInviteToken`) → `422` `invalid_registration_body`.
- Duplicate normalized email → `409` `email_taken`.
- **Email shape:** invalid format (after normalize) → **`422`** per OpenAPI/spec (JSON body validation tier), not `401`.
- **Login:** if `User.passwordHash` is **null** (OAuth-only placeholder), **reject password login** with the **same** `401` + `invalid_credentials` message as wrong password (no enumeration).
- Join path: optional **length/format check** on plaintext invite token **before** hash (spec: wrong length/format → **`422`**); then `sha256` (or `crypto.createHash('sha256')` in Bun) of token; lookup unused, unexpired invitation where **`tokenHash` matches**; wrong or expired → `422` `invite_invalid`; email mismatch → `422` `invite_email_mismatch`. Prefer **constant-length** token encoding; if comparing raw bytes in memory, use **`crypto.subtle.timingSafeEqual`** (spec: constant-time compare).
- Transaction: create `User` with `passwordHash`; create `Household` + membership `OWNER` OR membership `MEMBER` + mark invitation used; enforce **single membership** → `409` if user already has membership.

- [ ] **Step 1:** Unit tests with **mock Prisma** are optional; prefer **integration tests** in Task 12. Minimal: export pure helpers e.g. `assertRegistrationBodyShape(body): CreatePath | JoinPath | throws ApiProblem`.

- [ ] **Step 2:** Implement `register(...)` returning `{ accessToken, tokenType, expiresIn, user }` using `signAccessToken`. **`user`** must include **`id`**, **`email`**, **`displayName`** (spec: never expose `passwordHash`).

- [ ] **Step 3:** Implement `login(email, password)` — same envelope; wrong password or unknown email → throw `ApiProblem(401, 'invalid_credentials', 'Invalid email or password')` (one message).

- [ ] **Step 4:** Commit: `feat(api): auth register and login service`

### Task 7: HTTP handlers — register, login, logout

**Files:**

- Create: `src/api/handlers/auth.ts`
- Modify: `src/api/server.ts`
- Modify: `src/api/router.ts`

**Server flow (pseudocode):**

```ts
const path = url.pathname;
const method = req.method;
if (path === '/api/health') { ... }

if (path === '/api/auth/register' && method === 'POST')
  return handlePostRegister(req, db);
if (path === '/api/auth/login' && method === 'POST')
  return handlePostLogin(req, db);

// Wrong method on documented auth paths → 405 + Allow (same pattern as /api/health and router fallback)
const authPaths = ['/api/auth/register', '/api/auth/login', '/api/auth/logout'] as const;
if (authPaths.includes(path as (typeof authPaths)[number])) {
  const allow = apiAllowedMethodsForPathname(path);
  if (allow && !allow.includes(method)) {
    return new Response(null, { status: 405, headers: { Allow: allow.join(', ') } });
  }
}

const auth = resolveAuthUserId(req);
if (auth instanceof Response) return auth;

if (path === '/api/auth/logout' && method === 'POST')
  return handlePostLogout(); // 204 No Content

const { householdId } = await getHouseholdForUser(db, auth);
return dispatchApi(req, db, { prisma: db, userId: auth, householdId });
```

- [ ] **Step 2:** Implement `handlePostLogout`: accept empty body or `{}`; return **`new Response(null, { status: 204 })`** (spec preferred).

- [ ] **Step 3:** In `router.ts`, extend **`apiAllowedMethodsForPathname`** so `/api/auth/register`, `/api/auth/login`, and `/api/auth/logout` each return **`['POST']`** (required for the **`405`** branch in the pseudocode above). Add **`/api/household/invitations` → `['POST']`** when that route exists (Task 8).
